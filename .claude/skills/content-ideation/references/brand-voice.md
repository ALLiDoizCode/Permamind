# Brand Voice Guidelines - Permamind

## Core Identity
- **Industry**: Web3, Decentralized AI, Arweave/AO Ecosystem
- **Target Audience**: Developers, AI researchers, Web3 builders, early adopters
- **Brand Archetype**: The Innovator + The Sage
- **Mission**: Making AI memory permanent, decentralized, and accessible

## Tone Attributes

### Technical Depth with Accessibility (80/20)
- Deep technical accuracy for developers
- Accessible explanations for concepts
- Code examples and practical implementations
- Avoid dumbing down, but explain jargon

### Forward-Thinking but Grounded (70/30)
- Vision for decentralized AI future
- Realistic about current limitations
- Honest about trade-offs and challenges
- Data and evidence-based claims

### Community-Driven and Open (90/10)
- Open-source ethos and transparency
- Collaborative tone ("we're building together")
- Credit to community contributions
- Honest about what we don't know

### Educational and Empowering (75/25)
- Teach, don't just tell
- Provide context and background
- Empower builders to create
- Share learnings and failures

## Language Patterns

### Do:
- Use "we" and "our" (collaborative community)
- Provide concrete code examples and implementations
- Reference specific AO/Arweave concepts accurately
- Acknowledge complexity and nuance
- Share both successes and failures
- Link to documentation and resources
- Use technical terms correctly (process, handler, message, ADP)
- Ask thought-provoking questions
- Highlight practical use cases
- Give credit to other projects and contributors

### Don't:
- Use hype or buzzwords without substance ("revolutionary", "game-changing")
- Make absolute claims without evidence
- Oversimplify blockchain/AI concepts
- Be promotional or sales-focused
- Use Web3 jargon without explanation for newcomers
- Promise what we can't deliver
- Ignore limitations or challenges
- Take credit for others' work
- Use "crypto bro" or "to the moon" language

## Platform Adaptations

### Twitter/X
- **Tone**: Slightly more casual, timely, conversational
- **Format**: Threads for technical deep-dives, single tweets for updates
- **Engagement**: Ask questions, share learnings, highlight community
- **Examples**: "Just shipped: permanent memory for Claude agents. Here's what we learned building it üßµ"

### LinkedIn
- **Tone**: More professional, strategic insights, thought leadership
- **Format**: Long-form posts (1,300-1,900 chars), data-driven
- **Engagement**: Industry trends, case studies, technical architecture
- **Examples**: "The intersection of AI and decentralized storage: 3 architectural patterns we're seeing emerge..."

### Blog/Documentation
- **Tone**: Most technical, comprehensive, tutorial-focused
- **Format**: Long-form (1,500-3,000 words), code examples, diagrams
- **Engagement**: Deep dives, implementation guides, case studies
- **Examples**: "Building ADP-Compliant AO Processes: A Complete Guide"

### Discord/Community
- **Tone**: Helpful, authentic, peer-to-peer
- **Format**: Conversational, quick responses, code snippets
- **Engagement**: Answer questions, share updates, gather feedback
- **Examples**: "Good question! Here's how we handle that edge case in the MCP server..."

## Technical Terminology (Always Use Correctly)

### AO/Arweave Ecosystem:
- **AO Process**: Not "smart contract" or "program"
- **Handler**: Not "function" or "endpoint"
- **Message**: Not "transaction" or "call"
- **Permaweb**: Not "blockchain" or "Web3 storage"
- **ADP**: AO Documentation Protocol (always explain first use)

### Permamind Specific:
- **Permanent Memory**: Not "blockchain memory" or "decentralized storage"
- **MCP Server**: Model Context Protocol server (explain acronym)
- **Agent Skills**: Claude Code agent skills (not "plugins" or "extensions")
- **Memory Indexing**: Not "database" or "storage system"

5. Reflections and learnings

## Voice Checklist

Before publishing, verify:
- [ ] Technical accuracy (all AO/Arweave terms correct?)
- [ ] Accessibility (jargon explained for newcomers?)
- [ ] Evidence-based (claims supported by data/examples?)
- [ ] Collaborative tone (uses "we", gives credit?)
- [ ] Practical value (actionable insights or code?)
- [ ] Honest about limitations (acknowledges trade-offs?)
- [ ] Community-focused (empowers builders?)
- [ ] Free from hype (substance over buzzwords?)

## Readability Targets

- **Flesch Reading Ease**: 50-70 (fairly difficult to standard)
- **Grade Level**: 10-12 (technical audience)
- **Sentence Length**: 15-25 words average (mix short and long)
- **Paragraph Length**: 3-5 sentences (scannable)
- **Code-to-Text Ratio**: 20-40% for technical posts
